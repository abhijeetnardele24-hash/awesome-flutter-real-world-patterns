# Contributing

Thanks for contributing.

## What fits this repo

Good contributions usually have these qualities:

- solve one real Flutter problem
- use understandable structure
- avoid unnecessary abstraction
- include a short explanation of the tradeoffs
- feel reusable in real apps

## What to avoid

- toy examples with no practical value
- tutorial-style clutter
- giant demo apps for a small pattern
- generated or unreviewed code dumps

## Suggested contribution format

Each pattern should include:

- a clear folder name
- a short README
- the problem it solves
- the architecture decision
- the key files to inspect first
- notes about tradeoffs or limitations

## Naming

Prefer names like:

- `auth_repository_pattern`
- `offline_sync_queue`
- `file_upload_flow`

Avoid vague names like:

- `demo1`
- `final-app`
- `new-feature-test`

## Pull request checklist

- the example solves one clear problem
- the README is short and useful
- the structure is clean
- the code is understandable without extra explanation
